## Cross-project lessons captured (cross-tenant patterns)

1. **Meta API `Invalid parameter` is meta-vague.** Always extract `error_user_msg` / `error_user_title` / `error_subcode` from the response. Reproduce via curl + `python3 -m json.tool`. Two examples burned into us today: subcode 4834011 (`is_adset_budget_sharing_enabled` required) and subcode 1487793 (`Schedule Too Short` for daily_budget + 24h-exact). Filed.
2. **FB link cards are snapshotted per-post.** Force-scraping a URL via Sharing Debugger updates FB's URL cache for FUTURE posts but doesn't refresh the link card on existing posts. Solution: pre-scrape from the publishing code RIGHT BEFORE the `/feed` call, so the snapshot uses fresh OG. Now wired in `postToFacebook`. Pattern carries to any external publishing API that uses link cards (Twitter, LinkedIn, etc.).
3. **iMessage / SMS apps have their own OG cache** separate from FB's. Cache-bust query params if iMessage staleness becomes a recurring complaint; otherwise just trust that fresh viewers (who haven't visited the page before) get fresh OG.
4. **Stripe Issuing US requires individual cardholders** (Cross River Bank KYC). Card nicknames carry business-purpose labels. The cardholder is the human, the cards are the business uses. Self-funding loop: tenant Stripe balance (booking revenue) → Stripe Issuing card → ad spend → drives more bookings.
5. **Live data > static copy on feature pages.** /birding's BirdCast radar tile (live S3 fetch every 10 min) makes the page evergreen + always-fresh. Same pattern can apply to any tenant feature page that has a public data source — fishing-report could pull NOAA tide / surf data, conditions could pull weather, etc.
